# FastAPI + Celery 架構下的穩定性與資料一致性策略

在將「智慧檔案整理助理」從同步的 Streamlit 升級到非同步的 FastAPI + Celery 架構時，必須主動設計機制來應對併發處理帶來的挑戰。以下是確保系統穩定性與資料一致性的四大核心策略。

---

## 1. 確保任務的冪等性 (Idempotency)

**問題:** Celery 任務可能會因為網路問題、Worker 崩潰等原因被重複執行。如果檔案處理任務不是冪等的，重複執行會導致檔案被重複移動、資料庫紀錄重複或錯誤。

**策略:**

| 策略名稱 | 實作細節 | 確保一致性 |
| :--- | :--- | :--- |
| **檔案 Hash 檢查** | 在任務開始時，重新計算或驗證檔案的 SHA256 Hash。如果 Hash 已存在於資料庫中，則任務應立即終止 (Fail Fast)，避免重複處理。 | **防止重複處理**：確保每個檔案只被處理一次，避免檔案系統和資料庫的冗餘操作。 |
| **條件式檔案移動** | 在 `storage.py` 的 `organize_file` 邏輯中，使用原子操作（例如：檢查目標檔案是否存在，或使用資料庫鎖）來確保移動操作的唯一性。 | **防止檔案衝突**：避免兩個併發任務嘗試移動同一個檔案到同一個位置。 |
| **資料庫唯一索引** | 在 `Files` 表中，為 `file_hash` 欄位建立唯一索引 (Unique Index)。如果嘗試插入重複的 Hash，資料庫會拋出錯誤，確保資料層的一致性。 | **資料層唯一性**：利用資料庫的 ACID 特性，強制資料的唯一性。 |

---

## 2. 資源隔離與限制 (Resource Isolation & Throttling)

**問題:** 檔案處理，特別是 PDF 文字提取和潛在的 OCR 處理，是 CPU 密集型操作。如果 Celery Worker 無限制地併發執行，可能導致系統資源耗盡，影響整體穩定性。

**策略:**

| 策略名稱 | 實作細節 | 確保穩定性 |
| :--- | :--- | :--- |
| **任務分級與佇列** | 根據任務的資源消耗程度劃分 Celery 佇列。例如：`cpu_intensive` (PDF/OCR) 和 `io_intensive` (檔案移動/DB 寫入)。 | **資源隔離**：確保 CPU 密集型任務不會阻塞 I/O 密集型任務，提高系統響應性。 |
| **Worker 併發限制** | 使用 Celery 的 `-c` 參數限制每個 Worker 的併發數量。例如：`celery -A proj worker -c 4`。 | **資源控制**：防止 Worker 啟動過多進程，導致 CPU 或記憶體超載。 |
| **任務超時機制** | 為 Celery 任務設定 `time_limit` 和 `soft_time_limit`。如果任務執行時間過長，Celery 會自動終止該任務，防止無窮迴圈或資源洩漏。 | **防止資源洩漏**：確保異常任務不會長時間佔用系統資源。 |

---

## 3. 事務管理與狀態機 (Transaction Management & State Machine)

**問題:** 檔案處理是一個多步驟流程（提取 -> 分類 -> DB 寫入 -> 檔案移動）。任何一個步驟失敗都可能導致資料庫狀態與檔案系統狀態不一致。

**策略:**

| 策略名稱 | 實作細節 | 確保一致性 |
| :--- | :--- | :--- |
| **資料庫事務 (ACID)** | 將所有資料庫操作（例如：插入 `Files` 紀錄、更新 `FileTags`）包裹在單一的資料庫事務中。只有當所有操作都成功時才提交 (Commit)。 | **原子性 (Atomicity)**：確保資料庫操作要麼全部成功，要麼全部失敗，防止部分更新。 |
| **狀態機模型** | 在 `Files` 表中引入 `status` 欄位，定義明確的狀態轉換：`PENDING` -> `PROCESSING` -> `SUCCESS` / `FAILED`。 | **可追蹤性**：明確標記檔案的處理進度，方便監控和失敗重試。 |
| **兩階段提交 (Two-Phase Commit, 簡化)** | 雖然不使用真正的 2PC，但可以將檔案移動視為最後一步。先完成所有 DB 寫入，再執行檔案移動。如果檔案移動失敗，DB 狀態仍可標記為 `MOVE_FAILED`，方便人工介入或自動重試。 | **故障恢復**：將最容易失敗的 I/O 操作放在事務的邊緣，便於回滾或恢復。 |

---

## 4. 監控、日誌與錯誤處理 (Monitoring, Logging & Error Handling)

**問題:** 在非同步環境中，錯誤不會立即回饋給使用者，需要強大的監控系統來確保問題被及時發現和處理。

**策略:**

| 策略名稱 | 實作細節 | 確保穩定性 |
| :--- | :--- | :--- |
| **結構化日誌** | 使用 Python 的 `logging` 模組，並輸出結構化日誌 (例如 JSON 格式)，包含 `task_id`、`file_hash`、`timestamp` 等關鍵資訊。 | **可偵錯性**：方便使用 ELK Stack 或 Grafana Loki 等工具進行日誌分析和錯誤追蹤。 |
| **Celery 錯誤處理** | 實作 Celery 任務的 `on_failure` 處理函式，當任務失敗時，自動將 `Files` 表中的 `status` 更新為 `FAILED`，並記錄錯誤訊息。 | **自動化錯誤標記**：確保資料庫狀態能反映任務的真實執行結果。 |
| **重試機制** | 為 Celery 任務設定合理的自動重試策略 (`max_retries` 和 `retry_delay`)。例如，對於暫時性的網路錯誤，可以自動重試 3 次。 | **韌性 (Resilience)**：提高系統對暫時性故障的容忍度。 |

---

## 總結

通過實施上述策略，特別是**檔案 Hash 檢查**、**Celery 併發限制**和**資料庫事務**，我們可以將「智慧檔案整理助理」的架構提升到一個更專業的水平，使其能夠穩定、可靠地處理大量檔案，同時確保資料庫與檔案系統之間的高度一致性。這將是你在作品集中展示系統架構能力的有力證明。
