# 智慧檔案整理助理 (Smart File Organizer) 進階優化建議報告

**目標:** 將 MVP 升級為更具產品化潛力、技術更穩健的 V1 版本。

---

## 1. 技術架構優化：從 Streamlit 到 FastAPI + Celery

MVP 採用 Streamlit 快速實現 UI 和邏輯，但其本質是單執行緒、同步處理。當檔案數量增加時，會導致 UI 阻塞，使用者體驗極差。

### 1.1 核心問題：同步處理與 UI 阻塞

目前所有檔案處理（提取、分類、移動）都在 Streamlit 的主執行緒中完成。這使得使用者在等待處理時無法進行任何操作，且容易因超時而中斷。

### 1.2 建議優化方案：引入非同步任務佇列

| 優化項目 | 建議技術 | 說明 |
| :--- | :--- | :--- |
| **前後端分離** | **FastAPI + Vue/React** | 將 Streamlit 替換為 FastAPI 作為後端 API，提供標準化介面。前端使用 Vue/React 實現更複雜的互動和響應式設計。 |
| **非同步處理** | **Celery (或 Dramatiq)** | 引入任務佇列。使用者上傳檔案後，FastAPI 僅將任務丟入佇列，立即返回。Celery Worker 在背景執行耗時的檔案處理 Pipeline。 |
| **進度追蹤** | **Redis/RabbitMQ** | 用於 Celery 任務的 Broker 和 Backend，實現即時追蹤每個檔案的處理進度，並透過 WebSocket 或 Long Polling 推送到前端。 |

**優化後的處理流程 (V1 Pipeline):**

1.  使用者上傳檔案 (FastAPI 接收)。
2.  FastAPI 將檔案暫存，並將 `file_id` 丟給 Celery 任務佇列。
3.  FastAPI 立即返回 `Task ID` 給前端。
4.  前端使用 `Task ID` 輪詢 FastAPI 的 `/status/{task_id}` 接口。
5.  Celery Worker 執行 `core.py` 和 `storage.py` 的邏輯。
6.  Worker 完成後，更新 SQLite/PostgreSQL 中的狀態。
7.  前端顯示即時進度條，完成後顯示結果。

## 2. 核心模組優化：提升分類準確度與穩定性

目前的 Rule-based 分類器雖然可解釋，但面對多樣化的文件內容時，準確度會受限。

### 2.1 引入機器學習分類器 (V1 核心升級)

| 模組 | 現狀 (MVP) | 建議優化 (V1) | 效益 |
| :--- | :--- | :--- | :--- |
| **文件分類** | Rule-based (關鍵字打分) | **TF-IDF + Logistic Regression** | 實現多標籤分類，準確度更高，且模型輕量、可解釋。 |
| **照片分類** | 檔名規則 | **引入輕量級圖像標籤** | 使用預訓練的圖像分類模型 (如 MobileNetV2 或 ResNet 的輕量版本) 提取照片特徵，判斷是否為「風景」、「人物」、「食物」等，輔助分類。 |
| **文字提取** | 僅 `pypdf` | **整合 OCR 服務** | 引入 `Tesseract` 或雲端 OCR API (如 Google Vision API) 處理掃描檔，解決目前 MVP 無法處理掃描檔內容的問題。 |

### 2.2 強化時間提取邏輯

目前僅使用 EXIF 和檔案修改時間。優化方向應加入**日期解析**：

*   **文件日期解析:** 使用正則表達式或 NLP 技術，從 PDF 提取的文本中識別出最可能的日期（例如：合約簽署日期、發票日期），並將其作為最高優先級的日期。
*   **時區處理:** 檔案時間應考慮時區問題，確保日期標準化的一致性。

## 3. 資料庫與資料結構優化

### 3.1 實現多標籤與標籤管理

目前的 `Files` 表只有 `main_topic`，無法實現多標籤。

*   **實作 `Tags` 與 `FileTags` 關聯表**：這在原規劃中已提出，是 V1 必須完成的。這將允許一個檔案同時擁有 `發票` 和 `掃描` 兩個標籤。
*   **標籤管理介面**：在 UI 中增加一個介面，允許使用者自定義標籤、合併標籤或刪除標籤，讓系統更具彈性。

### 3.2 引入全文檢索 (Full-Text Search, FTS)

目前 SQLite 只能對 `original_name` 或 `main_topic` 進行簡單查詢。

*   **優化方案:** 利用 SQLite 內建的 **FTS5 模組**，對 `extracted_text` 欄位建立全文索引。
*   **效益:** 使用者可以輸入文件內容中的任意關鍵字（例如：「XYZ 公司」、「軟體開發」），系統即可快速找出相關文件，極大提升搜尋功能。

## 4. 使用者體驗 (UX) 優化

| 優化項目 | 說明 | 效益 |
| :--- | :--- | :--- |
| **結果確認機制** | 在檔案移動前，提供一個**「批次確認」**介面，允許使用者對系統判斷的日期、主題進行微調，避免錯誤整理。 | 提升使用者對系統的信任感和控制感。 |
| **重複檔案偵測** | 根據檔案的 SHA256 Hash (`file_hash` 欄位)，在上傳時即時提醒使用者檔案重複，並提供跳過或覆蓋的選項。 | 避免儲存庫中出現冗餘檔案。 |
| **錯誤處理與日誌** | 建立詳細的錯誤日誌系統，記錄處理失敗的檔案及其原因（例如：PDF 損壞、EXIF 讀取失敗），並在 UI 中清晰展示。 | 提升系統的健壯性與可維護性。 |

---

## 總結

MVP 專注於 **功能實現**，而 V1 則應專注於 **穩定性、準確度與使用者體驗**。

| 階段 | 核心目標 | 關鍵技術升級 |
| :--- | :--- | :--- |
| **MVP (已完成)** | 驗證核心 Pipeline | Python, Streamlit, SQLite, Rule-based |
| **V1 (建議升級)** | 提升產品穩定性與準確度 | FastAPI, Celery, TF-IDF/LR, FTS5, 多標籤 |

這些優化方向可以作為你未來 2-4 週的開發目標，讓你的 Side Project 真正具備產品級的品質。
